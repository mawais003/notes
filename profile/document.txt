Q6: What is the difference between Point-in-Time Recovery and Replication? Why do we need both?

"So replication is your proactive safety net — you maintain a live copy of the database that's continuously synced. If the primary goes down due to hardware failure or an AZ outage, the replica takes over with near-zero data loss. In PostgreSQL that's streaming replication using WAL segments.

Point-in-Time Recovery is different — it's for when the data itself gets corrupted. Say someone runs a DELETE without a WHERE clause, or a bad migration wipes a table. Replication can't save you there because the bad write gets replicated instantly to all replicas. PITR lets you restore to the exact second before that happened by replaying WAL files up to that specific timestamp.

So you need both — replication protects against infrastructure failures, PITR protects against logical or human errors. They cover completely different failure modes."

Q7: What is the difference between StatefulSet, DaemonSet, and Deployment?

"A Deployment is for stateless stuff — your APIs, web frontends. Pods are interchangeable, you kill one, another spins up, no state lost. Rolling updates and rollbacks are straightforward.

StatefulSet is for workloads that need identity and persistent storage. Each pod gets a fixed name — like kafka-0, kafka-1 — and its own PersistentVolumeClaim that survives restarts. They start and stop in order, which matters for things like cluster coordination.

DaemonSet ensures exactly one pod runs on every node in the cluster. When a new node joins, the DaemonSet automatically schedules a pod on it. Used for node-level infrastructure — log collection with Fluentd, metrics with node-exporter, network plugins."

Q8: Kafka, MCP server, and a collector — which controller for each? Why StatefulSet for Kafka?

"Kafka as a StatefulSet, MCP server as a Deployment, collector as a DaemonSet.

Kafka needs StatefulSet because each broker maintains partition data on local disk and registers with ZooKeeper or KRaft using a stable broker ID. If a broker restarts with a different identity or loses its storage, the cluster has to do expensive partition reassignment and rebalancing. StatefulSet guarantees stable network identity, ordered startup, and persistent volumes per pod — exactly what Kafka needs to maintain data consistency.

MCP server is stateless — any replica can handle any request, so a Deployment gives you easy horizontal scaling and rolling updates.

Collector needs to run on every single node to capture local logs or metrics, so DaemonSet is the natural fit."

Q9: What are the two technologies for automatic scaling of nodes in an EKS cluster?

"The two are Cluster Autoscaler and Karpenter.

Cluster Autoscaler is the older, traditional approach. It watches the Kubernetes scheduler for pods stuck in Pending state, then scales up the underlying Auto Scaling Group to add nodes. The limitation is it's reactive and slow — it polls every 10 to 15 seconds, then the ASG launches an instance from a fixed launch template, the node bootstraps and joins the cluster — the whole cycle can take 2 to 3 minutes. And you're locked into whatever instance types you pre-configured in the ASG.

Karpenter is the newer approach and what I use. It bypasses ASGs entirely — when pods can't be scheduled, Karpenter evaluates their actual resource requirements and calls the EC2 Fleet API directly to provision the right-sized instance. You define constraints in a NodePool CRD — instance families, AZs, Spot vs On-Demand. It's much faster, usually under 60 seconds. It also does automatic node consolidation — moves workloads to fewer nodes when demand drops and terminates the extras. For our streaming workloads with unpredictable spikes, Karpenter made a real difference."

If he asks about ECS instead:

"For ECS, the two approaches are ECS Capacity Providers with managed scaling, and raw ASG scaling policies. Capacity Providers are the smarter option — they track a metric called CapacityProviderReservation, which is the ratio of needed capacity versus available capacity, and scale the ASG based on actual task placement demand. They also have managed termination protection so ECS won't kill instances that still have running tasks. The alternative is configuring the ASG directly with CloudWatch-based scaling policies on CPU or memory — but for memory you need the CloudWatch Agent installed since AWS doesn't collect it by default."

Q10: After the RCA is written, what follow-up activities should we do?

"The RCA is only useful if it drives action. So first, every finding gets a permanent fix assigned to a specific owner with a deadline — not 'we should fix this eventually' but an actual ticket with a due date. Second, close the monitoring gap — if our alerting didn't catch the incident early enough, add the specific alerts that would have detected it. Third, update the runbooks with this new failure mode and the remediation steps so the next person on call handles it faster. Fourth, share it with the team — we do post-incident reviews so everyone learns from it, not just the two people who worked the incident. And finally, after the fixes are deployed, validate that they actually prevent recurrence — either through testing or by monitoring for the same pattern."
